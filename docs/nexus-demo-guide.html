Nexus — Demo Guide

A step-by-step walkthrough script for live demonstrations

Confidential
💡

Before you start: Have the app open to the home screen (Inventory tab), signed in on your site. Make sure you have a live internet connection — real-time sync is one of the best things to show. The whole demo should take 5–8 minutes. Don't rush. Let them read the screen.

Demo flow — walk through in this order
1
Open the About page first
Settings tab → "About Nexus" button
"Before I show you what it does, here's a quick overview of what we built." Let them read the screen for 10–15 seconds. Point out the 8 modules and the stat bar. "Everything you see here is live in production on real devices."
2
Inventory — the core
Tap the Inventory tab → open any item
"Every item has a current stock level, a minimum threshold, and a status — OK, Low, or Out. When stock drops below the minimum, the system automatically pushes a notification to every device on that site. No manual checking required."
3
Alerts — show the notification system
Tap the Alerts tab → show the Alerts view, then switch to Activity
"This is every stock event that's happened across the site — restocks, low stock, out of stock. There's a full activity log with date filters and you can export it to CSV. The Analytics view breaks down what's moving most and what gets alerted most often."
4
PM Checklist — the maintenance module
Tap the PM tab → open any device
"Every device has a full 30-point maintenance checklist — boot, login, network, hardware, security, cleaning, peripherals. Checks save instantly. If a device is overdue for its quarterly PM, it shows an orange badge right on the list. On the first of every month the system sends a push notification to remind the team how many devices are due."
5
Show real-time sync (if two devices available)
Adjust a stock quantity on one device, watch the other update
"Watch this — I'm going to change a quantity here." Change a number. Point to the second device. "It updated instantly. No refresh, no polling. Both devices are always looking at the same data."
6
Close with the security model
Mention it — no need to navigate anywhere
"One thing that's easy to overlook — every piece of data is locked to its site server-side. A staff member at Site A cannot see Site B's inventory even if they tried to query it directly. Admins manage users and sites. Staff can only see and edit their own site's data."
Key talking points
Say these with confidence
Built on the same stack as large consumer apps — React Native, Firebase, cloud functions
Deployed to real devices via EAS — updates ship without going through the App Store
Push notifications are end-to-end: database change → cloud function → device
Everything is multi-site — add a new location and it's fully isolated automatically
Dark and light mode, works on iOS and Android from one codebase
All security rules are enforced server-side — not just hidden in the UI
Likely questions & answers
How long did this take to build?
Built iteratively — started with inventory, added modules as needs came up. It's been running in production for months.
Can it support more sites?
Yes — the architecture is multi-site by design. Adding a new location is just adding a new site record. No code changes needed.
What happens if there's no internet?
Firestore caches recent reads locally, so you can still view data. Writes queue and sync when you reconnect.
Who can access it?
Only users with an account. Admins control who gets access and what site they belong to. Staff can't see other sites' data.
Can we add new features?
Yes — the codebase is modular and well-structured. New screens and features can be added without touching existing ones.
🎯

How to close the demo: "What you're looking at is a fully deployed, production application. Real data, real notifications, real users. This isn't a prototype — it's running right now." Then stop talking and let them ask questions. Silence after a strong close is good.
